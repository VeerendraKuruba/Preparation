# Airbnb Design Language System (DLS) — frontend system design

> **Scope:** Airbnb Design Language System as a real-world case study. For the full design system / frontend platform question, see [Q9 — Design System](./q09-design-system-frontend-platform.md).

Use this doc when the prompt touches **design systems at scale**, **consistency across web + native**, or **how a household brand operationalizes UI** (even if you are not building a copy of DLS).

---

## One-line definition

Airbnb's **Design Language System (DLS)** is an **internal, multi-platform design system**: shared **principles, tokens, components, and patterns** so product teams ship **coherent, accessible, branded** experiences on web, iOS, and Android.

- **Platforms:** Web only, or web + iOS + Android from a shared token source?

- **Maturity:** Greenfield system vs. migrating an existing codebase?

**Non-functional:** Bundle size per component; accessibility baseline (WCAG 2.1 AA minimum); theming (light/dark, brand variants); breaking change policy.

## 2. High-level architecture

```
┌─────────────────────────────────────────────────────────────────┐
│                   Design (Figma)                                │
│  Design tokens defined as Figma variables / token files         │
└──────────────────────────┬──────────────────────────────────────┘
                           │  Token JSON (color, spacing, radius…)
                           ▼
┌──────────────────────────────────────────────────────────────────┐
│              Style Dictionary (build pipeline)                   │
│  JSON → CSS custom properties, iOS Swift constants,              │
│         Android XML/Compose tokens, JS/TS constants              │
└──────┬──────────────────┬──────────────────┬─────────────────────┘
       │                  │                  │
       ▼                  ▼                  ▼
  @dls/tokens         @dls/ios-tokens    @dls/android-tokens
  (CSS vars + JS)     (Swift)            (Compose/XML)
       │
       ▼
  @dls/react           @dls/icons         @dls/motion
  (React components)   (SVG sprite)       (animation consts)
       │
       ▼
┌──────────────────────────────────────────────────────────────────┐
│             Product Apps (web)                                   │
│  Listings · Checkout · Host tools · Marketing pages             │
└──────────────────────────────────────────────────────────────────┘
       │
       ▼
  Docs site (Storybook / custom portal)
  Visual regression (Chromatic)
  Usage analytics (AST-based component scan)
```

| Layer | Role |
|-------|------|
| **Design tokens** | Single source of truth for color, spacing, typography, radius, elevation, motion. Platform-specific formats generated by Style Dictionary. |
| **Component library** | React (web), Swift (iOS), Compose (Android) — all honoring the same token contract. |
| **Documentation** | Storybook stories + written usage guidelines, do/don't, a11y notes, code examples. |
| **Governance** | RFC process, visual regression, semver, codemod tooling for breaking changes. |
| **Measurement** | Component usage tracking, bundle impact, % of product UI on DLS vs. bespoke CSS. |

### 3B. Semantic tokens — survive brand refresh without touching components

```
Semantic token  →  Primitive token  →  Raw value
─────────────────────────────────────────────────
color.action.primary  →  color.rausch.500  →  #FF5A5F

If Airbnb rebrands from Rausch red to a new accent color,
only the primitive token mapping changes.
Every component using color.action.primary updates automatically.
No component code changes required.
```

**Rule of thumb:** Components only ever reference **semantic** tokens. Primitive tokens exist only as the target of semantic token references. Product code should never hardcode `#FF5A5F` — that defeats the entire system.

### 3C. Dark mode — attribute-based token override, zero JS

```css
/* Dark mode: override semantic tokens under data-theme="dark" */
/* No JavaScript required — pure CSS cascade */
[data-theme="dark"] {
  --dls-color-background-primary: #222222;
  --dls-color-background-secondary: #383838;
  --dls-color-text-primary: #ffffff;
  --dls-color-text-secondary: #b0b0b0;
  --dls-color-action-primary: #FF5A5F; /* brand stays consistent */
}
```

```tsx
// Theme provider: reads system preference, stores override in localStorage
function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<'light' | 'dark'>(() => {
    const stored = localStorage.getItem('dls-theme');
    if (stored === 'light' || stored === 'dark') return stored;
    return window.matchMedia('(prefers-color-scheme: dark)').matches ? 'dark' : 'light';
  });

  useEffect(() => {
    document.documentElement.setAttribute('data-theme', theme);
    localStorage.setItem('dls-theme', theme);
  }, [theme]);

  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}
// Components need zero awareness of theme — they just use CSS variables.
// CSS variables resolve to the correct value based on the data-theme attribute.
```

### 3D. Component API — typed props, forwardRef, polymorphic "as" prop

```tsx
// DLS Button: variants, sizes, states — accessible by default
import { forwardRef, ComponentPropsWithoutRef, ElementType } from 'react';
import styles from './Button.module.css';

type ButtonVariant = 'primary' | 'secondary' | 'ghost' | 'destructive';
type ButtonSize = 'sm' | 'md' | 'lg';

// Polymorphic: can render as <button>, <a>, or any element
type PolymorphicProps<E extends ElementType> = {
  as?: E;
  variant?: ButtonVariant;
  size?: ButtonSize;
  loading?: boolean;
  fullWidth?: boolean;
} & Omit<ComponentPropsWithoutRef<E>, 'as'>;

function ButtonInner<E extends ElementType = 'button'>(
  {
    as,
    variant = 'primary',
    size = 'md',
    loading = false,
    fullWidth = false,
    disabled,
    children,
    className,
    ...rest
  }: PolymorphicProps<E>,
  ref: React.ForwardedRef<Element>
) {
  const Component = as ?? 'button';

  return (
    <Component
      ref={ref}
      className={[
        styles.button,
        styles[variant],
        styles[size],
        fullWidth && styles.fullWidth,
        loading && styles.loading,
        className,
      ]
        .filter(Boolean)
        .join(' ')}
      disabled={disabled || loading}
      // Accessibility: convey loading state to screen readers
      aria-busy={loading || undefined}
      aria-disabled={disabled || loading || undefined}
      {...rest}
    >
      {loading && <Spinner size={size} aria-hidden />}
      <span className={loading ? styles.loadingText : undefined}>{children}</span>
    </Component>
  );
}

export const Button = forwardRef(ButtonInner) as typeof ButtonInner;

// Usage examples:
// <Button variant="primary" size="lg">Book now</Button>
// <Button as="a" href="/listings" variant="ghost">Browse</Button>
// <Button variant="destructive" loading={isDeleting}>Delete listing</Button>
```

### 3E. Multi-platform — same token JSON, different platform implementations

```
Token JSON (single source of truth)
          │
          ├──▶ CSS custom properties  → React web components
          ├──▶ Swift enums/constants  → SwiftUI / UIKit components
          └──▶ Compose tokens / XML   → Jetpack Compose components

// iOS Swift output (generated by Style Dictionary):
public enum DLSColor {
  public static let backgroundPrimary = UIColor(hex: "#ffffff")
  public static let actionPrimary = UIColor(hex: "#FF5A5F")
}

// Android Compose output (generated by Style Dictionary):
object DLSColor {
  val BackgroundPrimary = Color(0xFFFFFFFF)
  val ActionPrimary = Color(0xFFFF5A5F)
}
```

The **contract** between platforms is the token names (semantic layer). The **implementation** is platform-appropriate — UIKit on iOS, Compose on Android, CSS variables on web. A product designer changing a token in Figma triggers a pipeline that regenerates all three platform outputs.

### 3F. Governance — RFC process, visual regression, breaking change policy

```
Contribution lifecycle:
─────────────────────────────────────────────────────
1. PROPOSE    → RFC document (GitHub Discussion or internal tool)
               Describes: problem, proposed API, alternatives, migration path
               Required for: new components, breaking API changes, new tokens

2. REVIEW     → DLS team + stakeholder review (2 approvals required)
               Design review: Figma mock aligns with spec
               Engineering review: API consistency, a11y, bundle impact

3. BUILD      → PR with:
               - Component implementation
               - Unit tests (RTL): renders, keyboard, ARIA attributes
               - Visual stories (Storybook): all variants × states × themes
               - Chromatic baseline (automated visual regression)

4. RELEASE    → Semver:
               - PATCH: bug fixes, no API change
               - MINOR: new variants/props, backward compatible
               - MAJOR: breaking prop rename/removal → must include codemod

5. COMMUNICATE → Changelog entry, Slack announcement, migration guide
─────────────────────────────────────────────────────

// Example codemod for breaking change (jscodeshift):
// Old: <Button type="cta">  →  New: <Button variant="primary">
module.exports = function transformer(file, api) {
  const j = api.jscodeshift;
  return j(file.source)
    .find(j.JSXOpeningElement, { name: { name: 'Button' } })
    .forEach((path) => {
      path.node.attributes.forEach((attr) => {
        if (attr.name?.name === 'type' && attr.value?.value === 'cta') {
          attr.name.name = 'variant';
          attr.value.value = 'primary';
        }
      });
    })
    .toSource();
};
// Run: npx jscodeshift -t button-type-to-variant.js src/**/*.tsx
```

### 3G. Adoption metrics — usage tracking, DLS coverage

```ts
// AST-based component usage scan (runs in CI or as a scheduled job)
// Counts how many times each DLS component is imported across product repos

import { Project } from 'ts-morph';

function scanDLSUsage(sourceGlob: string) {
  const project = new Project();
  project.addSourceFilesAtPaths(sourceGlob);

  const usageMap = new Map<string, number>();

  project.getSourceFiles().forEach((file) => {
    file.getImportDeclarations()
      .filter((imp) => imp.getModuleSpecifierValue().startsWith('@dls/'))
      .forEach((imp) => {
        imp.getNamedImports().forEach((named) => {
          const name = named.getName();
          usageMap.set(name, (usageMap.get(name) ?? 0) + 1);
        });
      });
  });

  return usageMap;
}
```

## 4. Publicly discussed principles

| Principle | Meaning for frontend / system design |
|-----------|--------------------------------------|
| **Unified** | Components and pages read as **one product**; fewer one-offs; shared tokens and patterns reduce drift. |
| **Universal** | **Global users** and **accessibility** (WCAG-minded baselines) are first-class — not a polish pass at the end. |
| **Iconic / distinctive** | Strong brand and recognizable patterns; avoids "generic admin template" sameness while still using systemized pieces. |
| **Conversational / intuitive** | Interaction design and clarity matter as much as visual polish — **copy, flow, and feedback states** are part of the system. |

## 5. Trade-offs

| Approach | Upside | Cost |
|----------|--------|------|
| **Monorepo (single repo)** | Atomic cross-package changes, unified CI, shared tooling | Repo grows large; requires invest in build caching (Turborepo/Nx) |
| **Multi-repo** | Teams are autonomous, independent release cadence | Versioning drift; consumers stuck on old packages; hard to refactor across |
| **CSS-in-JS (styled-components / Emotion)** | Co-located styles, dynamic theming, TypeScript prop checks | Runtime overhead, SSR complexity, large bundle, hard to extract static CSS |
| **CSS Modules** | Zero runtime, scoped by default, standard CSS | No runtime theming; dark mode via attribute + custom properties (our approach) |
| **Tailwind** | Rapid prototyping, low-overhead utility classes | Departs from semantic token layer; harder to enforce brand constraints |
| **Strict API (no escape hatches)** | Consistency, speed, accessibility | Slower for truly novel marketing or experiments |
| **Flexible "primitive + recipes"** | Faster product iteration | More ways teams misuse primitives; governance harder |
| **Central team owns everything** | Quality bar, coherent vision | Bottleneck if understaffed; contributions slow down at scale |
| **Federated contribution** | Scales with org size | Needs strong RFCs and automated design QA to avoid drift |

**DLS recommendation:** CSS Modules + CSS custom properties for web (zero runtime, works with SSR, dark mode free); centralized governance for foundation tokens and core components; federated contribution with RFC guard-rails for product-specific patterns.

## 6. Pitfalls (bonus interview points)

- **Orphan components:** Shipped once, never documented — become implicit API that nobody can safely deprecate. Solution: adoption tracking + explicit deprecation ceremony.
- **No deprecation policy:** Fear of breaking consumers freezes improvement. Solution: semver major + codemod + 2-sprint migration window.
- **Visual-only system:** Pretty Figma, weak behavioral spec (focus states, loading, error, empty states). Solution: Storybook stories for every interactive state; a11y checks in CI.
- **One bundle to rule them all:** Every page pays full cost. Solution: each component is its own entry point; consumers tree-shake; no barrel `index.ts` that pulls in everything.
- **Tokens without semantics:** Exposing only `color.blue.500` forces product engineers to make brand decisions. Semantic layer (`color.action.primary`) centralizes that decision in the design system.
- **Skipping mobile native:** Web DLS without iOS/Android counterpart means squads build bespoke native UI anyway. Shared token JSON is the minimal investment to create visual alignment.

## 7. Generalize: what you'd design without naming Airbnb

If the interviewer wants your own system, not a case study:

1. **Tokens + themes** (light/dark, brand variants) generated by Style Dictionary.
2. **Semantic token layer** so components survive brand refresh.
3. **Component API consistency** (composition, sizing variants, forwardRef, polymorphic `as`).
4. **Documentation** (Storybook stories, usage do/don't, a11y notes, code examples).
5. **Versioning and migration** (semver + codemod + migration guide for every major).
6. **Adoption incentives** (CLI scaffolds: `dls add Button`; lint rules: `no-custom-button`; Figma ↔ code parity checks).
7. **Measurement** (AST usage scan, bundle impact, WCAG audit score, design debt backlog).

## 10. Closing statement

"A design system like Airbnb's DLS is a **product for product teams** — its customers are engineers and designers, its deliverables are tokens, components, patterns, and the governance processes that keep them coherent. The architecture starts with **a single token JSON** (color, spacing, radius, motion) that Style Dictionary compiles into CSS custom properties for web, Swift constants for iOS, and Compose values for Android. Components reference only **semantic tokens** so a brand refresh never requires touching component code. On web, **CSS Modules + CSS custom properties** give zero-runtime theming including dark mode via a single `data-theme` attribute. Components expose typed, forward-ref props with a polymorphic `as` pattern and are backed by Storybook stories covering every interactive state and Chromatic visual regression. Governance runs through an RFC process: MINOR changes extend the API backwards-compatibly, MAJOR changes ship with a jscodeshift codemod. Adoption is measured by AST scanning across product repos, so the team can see which squads still use bespoke UI and target migration effort. The hardest part is not the technical implementation — it is the **organizational trust** built through documentation quality, fast review turnaround, and predictable versioning."
