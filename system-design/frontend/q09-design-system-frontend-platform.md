# Q9. Design System / Frontend Platform (Many Teams)

**Prompt variants:** Frontend at 50+ teams — how do UI and releases scale?

[← Question index](./README.md)

**Also:** [Micro frontends](../micro-frontends.md) · [Airbnb DLS](../airbnb-design-system.md)

---

### One-line mental model

**Contracts beat heroics**: versioned packages, tokens, examples, and governance so 50 product teams move fast without creating N conflicting button implementations — the platform team's job is to make the right thing the easy thing.

### Clarify scope in the interview

- Web only, or also React Native / mobile?
- Monorepo (Nx, Turborepo) or separate repos per package?
- Who approves breaking changes — a central platform team, or contributors from product teams?
- Release cadence: continuous delivery or monthly train?
- White-label / multi-brand theming required?
- Compliance: WCAG AA enforced in CI?

### Goals & requirements

**Functional:**
- Component library covering primitives (Button, Input, Modal, Table, etc.) and patterns (Form, DataTable, CommandPalette)
- Design token system: colors, typography, spacing, motion — single source of truth
- Documentation portal: live examples, prop tables, accessibility notes, design spec links
- Theming: default brand + overrides (dark mode, white-label)

**Non-functional:**
- Adoption: >80% of UI code imports from the design system, not rogue copies
- Semver discipline: product teams are never surprised by a breaking change
- Bundle size: tree-shakeable — importing one Button does not load the whole library
- Quality gates: visual regression, a11y, and bundle size checked on every PR
- MTTR for P0 component bugs < 24h

### What the client does (core mechanics)

#### 1. Design tokens — CSS variables as the foundation

Tokens are defined once in JSON (or Figma variables) and compiled to every format by Style Dictionary:

```json
// tokens/color.json — source of truth
{
  "color": {
    "brand": {
      "primary": { "value": "#0052CC" },
      "primary-hover": { "value": "#0747A6" }
    },
    "feedback": {
      "error": { "value": "#DE350B" },
      "success": { "value": "#006644" }
    },
    "neutral": {
      "0": { "value": "#FFFFFF" },
      "100": { "value": "#F4F5F7" },
      "900": { "value": "#091E42" }
    }
  }
}
```

```javascript
// style-dictionary.config.js — generates CSS vars + JS + SCSS
module.exports = {
  source: ['tokens/**/*.json'],
  platforms: {
    css: {
      transformGroup: 'css',
      prefix: 'corp',
      files: [{
        destination: 'dist/tokens.css',
        format: 'css/variables',
        // output: --corp-color-brand-primary: #0052CC;
      }]
    },
    js: {
      transformGroup: 'js',
      files: [{
        destination: 'dist/tokens.js',
        format: 'javascript/es6',
        // output: export const colorBrandPrimary = '#0052CC';
      }]
    }
  }
};
```

Components consume tokens via CSS variables — never hardcode hex values:

```css
/* Button.module.css */
.button-primary {
  background-color: var(--corp-color-brand-primary);
  color: var(--corp-color-neutral-0);
}
.button-primary:hover {
  background-color: var(--corp-color-brand-primary-hover);
}
```

Dark mode is a token override, not a CSS `prefers-color-scheme` one-liner:

```css
/* Apply at root — can be toggled by JS for user preference */
[data-theme="dark"] {
  --corp-color-brand-primary: #579DFF;
  --corp-color-neutral-0: #1D2125;
  --corp-color-neutral-900: #C7D1DB;
}
```

#### 2. Component API contracts — strict, documented, semver-governed

```typescript
// Button.types.ts — the public API contract
export interface ButtonProps {
  /** Primary action style; use sparingly — one per screen section */
  variant: 'primary' | 'secondary' | 'ghost' | 'danger';
  size?: 'sm' | 'md' | 'lg';
  /** Loading state: shows spinner, disables interaction, preserves width */
  isLoading?: boolean;
  /** Renders as an anchor tag when provided */
  href?: string;
  /** If href is provided, opens in new tab and adds screen-reader notice */
  external?: boolean;
  disabled?: boolean;
  onClick?: (event: React.MouseEvent<HTMLButtonElement>) => void;
  children: React.ReactNode;
  /** Forwarded to the underlying DOM element */
  className?: string;
  'aria-label'?: string;
}

// Button.tsx
export const Button = React.forwardRef<HTMLButtonElement, ButtonProps>(
  ({ variant, size = 'md', isLoading, href, external, children, ...rest }, ref) => {
    const Tag = href ? 'a' : 'button';

    return (
      <Tag
        ref={ref as any}
        className={clsx(styles.button, styles[variant], styles[size], {
          [styles.loading]: isLoading,
        })}
        disabled={isLoading || rest.disabled}
        aria-busy={isLoading}
        href={href}
        target={external ? '_blank' : undefined}
        rel={external ? 'noopener noreferrer' : undefined}
        {...rest}
      >
        {isLoading && <Spinner aria-hidden size="sm" />}
        <span className={isLoading ? styles.hidden : undefined}>{children}</span>
        {external && <span className={styles.srOnly}>(opens in new tab)</span>}
      </Tag>
    );
  }
);
```

**API contract rules:**
- Props are additive in minor versions. Removal or rename is a **major** version bump.
- Behavior changes (e.g., changing default `size`) are breaking — require RFC and codemod.
- Internal implementation (CSS class names, DOM structure) is not part of the contract and can change in any version.

#### 3. Versioning strategy and release process

```
Semver rules for the design system:

  MAJOR (2.0.0) — breaking changes to public API
    - Remove or rename a prop
    - Change a component's default behavior
    - Remove a component entirely
    - Token rename

  MINOR (2.1.0) — new backward-compatible features
    - Add a new optional prop
    - Add a new component
    - Add a new token

  PATCH (2.1.1) — bug fixes
    - Visual bug in existing behavior
    - Accessibility fix that doesn't break the API
    - Performance improvement

Release train:
  main branch → canary tag (nightly, auto-published)
    ↓  (canary app runs against it — internal dogfood app)
  PR + review → next tag (beta, teams can opt in)
    ↓  (1 week soak)
  Merge to release → stable tag  →  npm @corp/ui@2.14.0
```

#### 4. Visual regression testing with Chromatic

```yaml
# .github/workflows/chromatic.yml
- name: Run Chromatic
  uses: chromaui/action@v1
  with:
    projectToken: ${{ secrets.CHROMATIC_PROJECT_TOKEN }}
    exitZeroOnChanges: false  # fail PR if visual changes are unreviewed
    onlyChanged: true         # only test stories for changed components
    autoAcceptChanges: main   # auto-accept on merge to main (intentional)
```

Every Storybook story is a visual regression test. Chromatic snapshots each story in multiple viewports and browsers. Any pixel diff blocks the PR until a designer or platform team member reviews and approves.

```typescript
// Button.stories.tsx — each story = one visual snapshot
export const AllVariants: Story = {

    <div style={{ display: 'flex', gap: 8 }}>
```

#### 5. Tree-shaking and bundle size governance

```typescript
// Package structure — named exports, no barrel re-exports from one giant index
// @corp/ui/button  →  Button + ButtonProps
// @corp/ui/modal   →  Modal + ModalProps
// NOT: @corp/ui  →  { Button, Modal, Input, ... } (kills tree-shaking)

// package.json sideEffects field — tells bundlers this package is tree-shakeable
{
  "name": "@corp/ui",
  "sideEffects": ["**/*.css"],  // CSS has side effects; JS does not
  "exports": {
    "./button": {
      "import": "./dist/button/index.esm.js",
      "require": "./dist/button/index.cjs.js",
      "types": "./dist/button/index.d.ts"
    },
    "./modal": { ... }
  }
}
```

Bundle size CI gate:

```yaml
# bundlemon checks each component's gzipped size against a budget
# fails PR if any component grows beyond threshold
- name: BundleMon
  run: npx bundlemon
# .bundlemonrc.json
{
  "files": [
    { "path": "dist/button/index.esm.js", "maxSize": "4kb" },
    { "path": "dist/modal/index.esm.js",  "maxSize": "12kb" }
  ]
}
```

#### 6. Governance — RFC process for breaking changes

```
Breaking change lifecycle:

  1. PROPOSAL: engineer opens RFC issue with:
     - What changes and why
     - Migration path for existing consumers
     - Codemod availability (yes/no)
     - Target timeline (e.g., major release in 6 weeks)

  2. REVIEW: 1-week comment period, async
     Platform team + impacted team leads review

  3. CODEMOD: if prop rename/removal, write and test a jscodeshift codemod
     # npx @corp/codemods button-variant-rename
     # transforms: variant="default" → variant="secondary" in all consumers

  4. DEPRECATION PERIOD: mark old prop as deprecated in TypeScript + docs
     @deprecated Use variant="secondary" instead. Will be removed in v3.0.
     Show console.warn in development builds only.

  5. MAJOR RELEASE: remove deprecated API, publish v3.0.0

  6. MIGRATION SUPPORT: platform team offers migration office hours
     for the 2 weeks after the major release
```

### Trade-offs

| Decision | Monorepo | Separate repos | Recommendation |
|---|---|---|---|
| Developer experience | One PR touches tokens + component + docs | Context switching across repos | Monorepo wins for the platform team; use Nx or Turborepo for caching |
| Consumer isolation | Consumers share one yarn.lock — upgrades are coordinated | Each consumer upgrades independently | Monorepo for the DS itself; consumers are separate apps |
| CI cost | One large CI run; Turborepo caches unaffected packages | Small per-repo CI | Turborepo + remote cache makes monorepo viable |

| Decision | CSS-in-JS (styled-components / Emotion) | CSS Modules | Utility classes (Tailwind) |
|---|---|---|---|
| Runtime cost | Style injection at render time — adds ~10 kB runtime | Zero runtime — static CSS | Zero runtime |
| Theming | Dynamic via props — easy | CSS variables — easy | CSS variables — easy |
| Isolation | Automatic class name hashing | Hashed class names | Requires strict discipline |
| Extractability | Hard to extract to static CSS | Easy — output is regular CSS | Easy |
| Recommendation | Avoid in a shared library (runtime dep on consumers) | Best for a component library | Good for product apps, harder for a library with tokens |

**Verdict:** Use CSS Modules + CSS variables (tokens) for the component library. Consumers can use whatever they want in their app code.

| Decision | Storybook | Custom docs portal | Both |
|---|---|---|---|
| Effort | Medium — setup + maintenance | High — build from scratch | High |
| Live examples | Built-in | Custom implementation | — |
| VRT integration | First-class (Chromatic) | Custom | Use Storybook + Chromatic |
| Recommendation | Storybook for component development and VRT; build a lightweight marketing/docs site on top via `@storybook/addon-docs` | | |

### Failure modes & degradation

- **Bad release breaks a major product app:** Canary app catches it first (it always runs against the `next` tag). If it escapes to stable, the fix path is: publish a patch version immediately, send a comms message to #design-system-consumers Slack channel, auto-create a GitHub issue.
- **Version skew in MFE setups:** Two remotes consume `@corp/ui@2.x` and `@corp/ui@3.x` simultaneously. React singleton constraint means two React versions = crash. Solution: pin shared deps in Module Federation config; coordinate major version upgrades across remotes as a synchronized rollout.
- **Token rename breaks CSS:** If a CSS variable name changes (e.g., `--corp-color-primary` → `--corp-color-brand-primary`), products that hardcoded the old variable name break silently (no TypeScript error). Mitigation: CSS variable linting rule (stylelint plugin) that flags unknown `--corp-*` variables in product code.
- **Stale canary app:** If the canary app isn't maintained, it stops being a useful signal. Assign ownership to the platform team; make the canary app a first-class internal product.

### Accessibility checklist

- Components ship with a11y tests baked in — not added later. Use `@storybook/addon-a11y` (axe-core) on every story: zero violations required to merge.
- Focus trap in Modal, Drawer, Popover — tested with keyboard-only navigation in CI via Playwright.
- Color contrast: all token pairs checked at build time against WCAG AA (4.5:1 for text). A token PR that introduces a failing contrast pair fails CI.
- ARIA patterns: Button, Combobox, Dialog, Menu follow the ARIA Authoring Practices Guide patterns exactly — not approximations.
- Screen reader smoke tests: critical components (Modal, Combobox, Alert) have Playwright + VoiceOver/NVDA tests in the quality suite.

### Minute summary (closing)

"I'd scale the frontend platform with three layers: design tokens as the single source of truth compiled to CSS variables and JS constants, a versioned component library with strict semver and an RFC process for breaking changes, and a CI pipeline that gates every PR on visual regression, accessibility, and bundle size. Teams consuming the library never see surprise breaking changes — deprecated props come with codemods and a migration window. For organizations with multiple independent deploy teams, MFE boundaries via Module Federation can layer on top, but the component library and tokens remain shared dependencies. The measure of success is adoption rate — if product teams aren't using the system, they're building their own, which defeats the purpose."

- **Package graph:** `@corp/tokens` ← `@corp/ui` ← product apps + optional MFE remotes.

### Deep dives — pick 2

**A. Governance** — Walk through the RFC lifecycle. Emphasize that breaking changes need codemods — a codemod makes a breaking change non-breaking in practice for most consumers. Explain the deprecation window (warn in dev, not prod). Discuss who has merge rights on the DS repo: a platform team that reviews all PRs, or a contributor model with platform final approval.

**B. Micro-frontends** — When to introduce MFE boundaries: different release cadences, different tech stacks, team ownership. Module Federation shared deps config. The `singleton: true` flag for React. Explain version skew risk and the major-version coordination problem. Reference [micro-frontends.md](../micro-frontends.md).

**C. Quality gates** — Chromatic VRT in detail: how it works (screenshot diffing per story), how to handle intentional changes (approve in UI), how to avoid flaky tests (no animations in test mode). Axe integration in Storybook. Bundle size budgets per component. The cost of not having these: visual regressions caught by users, not CI.

**D. Theming & branding** — Token tiers: global tokens (raw values) → semantic tokens (role-based, e.g., `color-text-primary`) → component tokens (e.g., `button-background`). White-label overrides only need to change semantic tokens. Dark mode via `data-theme` attribute. Contrast budget automation.

### Common follow-ups

- **"Designers code?"** — The pipeline is: designers update variables in Figma → a GitHub Action pulls the Figma token API → opens a PR with the generated JSON diff → platform team reviews and merges → tokens publish. Designers don't write code, but they own the token values.
- **"Third-party widgets?"** — Isolate in a wrapper component that normalizes the API. Lazy-load the third-party bundle. Shadow DOM gives CSS isolation but blocks token inheritance (CSS variables don't pierce shadow DOM without explicit forwarding). Contractual: the wrapper owns the public API; the third-party is an implementation detail.
- **"Emergency hotfix on a primitive?"** — Patch version published within hours. Cherry-pick onto the stable release branch (not main, which may have unreleased changes). Notify consumers via Slack + GitHub release notes. If the bug is severe, offer an emergency override via a CSS variable injection shim while teams upgrade at their own pace.

### What you'd measure

- **Adoption:** percentage of UI surface area importing from `@corp/ui` vs rogue copies (detected via ESLint rule and code search). Target: >80%.
- **Health:** P0 component bug MTTR (time from report to patch release). Target: < 24h.
- **Quality:** VRT false positive rate (flaky Chromatic tests waste reviewer time). A11y violations caught in CI vs escaping to production (target: zero in production).
- **Bundle impact:** per-component gzipped size trend over time. Alert on any component growing > 20% in a single release.
- **Consumer velocity:** time from DS release to consumer adoption (measures how painful upgrades are — if it's > 4 weeks on average, your migration story needs work).
